【译】6.824 Lab 2: Raft

Posted on 2021-04-17

介绍

这是一系列实验中的第一个，你将在其中构建容错的键/值存储系统。在本实验中，你将实现 Raft（复制状态机协议）。在下一个实验中，你将在 Raft 之上构建键/值服务。然后，你将在多个复制的状态机（multiple replicated state machines）上“分片”服务，以提高性能。

复制服务通过将其状态（即数据）的完整副本存储在多个副本服务器上来实现容错功能。即使服务的某些服务器出现故障（崩溃，网络故障或不稳定），复制也可以使服务继续运行。挑战在于，故障可能导致副本持有不同的数据副本。

Raft 将客户请求组织成一个序列，称为日志（log），并确保所有副本服务器看到相同的日志。每个副本均以日志顺序执行客户端请求，并将其应用于服务状态的本地副本。由于所有活动副本都看到相同的日志内容，因此它们都以相同的顺序执行相同的请求，因此继续具有相同的服务状态。如果服务器出现故障但后来又恢复了，Raft 会确保其日志为最新状态。只要至少大多数服务器处于活动状态并且可以相互通信，Raft 就会继续运行。如果没有这样的多数，Raft 将不会取得进展，但是一旦多数能够再次交流，它将从停下来的地方继续前进。

在本实验中，你将通过 associated methods 将 Raft 实现为 Go 对象类型，该方法将用作更大服务中的模块。一组 Raft 实例通过 RPC 相互通信以维护复制的日志。你的 Raft 接口将支持不确定编号的命令序列，也称为日志条目。这些条目已用索引号编号。具有给定索引的日志条目最终将被提交。届时，你的 Raft 应该将日志条目发送到更大的服务，以便其执行。

你应该按照扩展的 Raft 论文中的设计进行操作，尤其要注意图 2。你将实现论文中的大部分内容，包括保存持久状态并在节点发生故障后读取它，然后重新启动。你将不会实现集群成员资格更改（第 6 节）。

你可能会发现本指南很有用，以及有关锁和并发结构的建议。从更广泛的角度来看，请查看 Paxos，Chubby，Paxos Made Live，Spanner，Zookeeper，Harp，Viewstamped Replication 和 Bolosky 等。 （注意：学生指南是几年前写的，尤其是 2D 部分已更改。请确保在盲目地遵循它之前理解为什么特定的实施策略有意义！）

GFS 论文笔记

Posted on 2021-04-17

GFS 是什么？

GFS 是一个分布式文件系统，并且具有如下特点：

  • 运行在普通的硬件设备集群上
  • 面向数据密集型应用（且 GFS 是以 google 内部的应用负载情况和技术环境分析为基础设计的）
  • 容错，可伸缩，高性能，高可靠

GFS 的设计假设

  • 组件失效被认为是常态事件，而不是意外事件
  • 以通常的标准衡量，我们的文件非常巨大
  • 绝大部分文件的修改是采用在文件尾部追加数据，而不是覆盖原有数据的方式
  • 应用程序和文件系统 API 的协同设计提高了整个系统的灵活性

GFS 的设计预期

  • 系统必须持续监控自身的状态，它必须将组件失效作为一种常态，能够迅速地侦测、冗余并恢复失效的组件
  • 主要针对大文件进行优化，但也支持小文件
  • 系统主要负载：大规模的流式读取和小规模的随机读取。对于小规模随机读取，采用合并排序后按序批量读取的方式
  • 系统负载还包括许多大规模的、顺序的、数据追加方式的写操作。系统也支持低效的小规模数据随机写入
  • 系统必须高效的、行为定义明确的2实现多客户端并行追加数据到同一个文件里的语意。使用最小的同步开销来实现的原子的多路追加数据操作是必不可少的。
  • 高性能的稳定网络带宽远比低延迟重要。

GFS 的接口

  • 类传统文件操作 API，但不满足 POSIX 标准
  • 文件以分层目录的形式组织，用路径名来标识
  • 常用文件操作：创建新文件、删除文件、打开文件、关闭文件、读和写文件
  • 快照，低的成本创建一个文件或者目录树的拷贝
  • 记录追加，允许多个客户端同时对一个文件进行数据追加操作，同时保证每个客户端的追加操作都是原子性的。

GFS 架构

  • 单 master，多 chunkserver，多 client
  • chunk
    • 固定大小 64 MB
    • 唯一不变的 chunk 标识
    • Chunk 服务器把 Chunk 以 Linux 文件的形式保存在本地硬盘上，并且根据指定的 Chunk 标识和字节范围来读写 chunk 数据
    • 缺省情况下，每个 chunk 会被复制到 3 个 chunkserver 上存储（用户可以为不同的文件命名空间设定不同的复制级别。）
  • master
    • 存储元数据
      • 名字空间、访问控制信息、文件和 Chunk 的映射信息、以及当前 Chunk 的位置信息
    • 管理系统范围内活动
      • Chunk 租用管理、孤儿 Chunk 的回收、以及 Chunk 在 Chunk 服务器之间的迁移
    • HeartBeat messages
      • 使用心跳信息周期地和每个 Chunk 服务器通讯，发送指令到各个 Chunk 服务器并接收 Chunk 服务器状态信息
  • 客户端
    • 代码
      • 以库的形式被链接到客户程序里
      • GFS 文件系统的 API 接口函数
      • 应用程序与 Master 节点和 Chunk 服务器通讯
      • 对数据进行读写操作
    • 客户端和 Master 节点的通信只获取元数据，所有的数据操作都是由客户端直接和 Chunk 服务器进行交互的
  • 数据缓存
    • 客户端只缓存元数据
    • chunkserver 中的 chunk 以本地文件的形式存储，Linux 操作系统会自动缓存

客户端的读取流程

  • 客户端把文件名和程序指定的字节偏移，根据固定的 Chunk 大小，转换成文件的 Chunk 索引
  • 把文件名和 Chunk 索引发送给 Master 节点
  • Master 节点将相应的 Chunk 标识和副本的位置信息发还给客户端
  • 客户端用文件名和 Chunk 索引作为 key 缓存这些信息
  • 客户端发送请求到其中的一个副本处，一般会选择最近的。请求信息包含了 Chunk 的标识和字节范围。

在对这个 Chunk 的后续读取操作中，客户端不必再和 Master 节点通讯了，除非缓存的元数据信息过期或者文件被重新打开。

实际上，客户端通常会在一次请求中查询多个 Chunk 信息，Master 节点的回应也可能包含了紧跟着这些被请求的 Chunk 后面的 Chunk 的信息，从而避免了客户端和 Master 节点未来可能会发生的几次通讯。

chunk 尺寸

  • 64 MB
  • 缺点：
    • 可能会存在内存碎片（即要求这么大的内存空间，事实上只用了一点点）。
      • 惰性空间分配策略避免了因内部碎片造成的空间浪费
    • 小文件包含较少的 Chunk，甚至只有一个 Chunk，当有许多的客户端对同一个小文件进行多次的访问时，存储这些 Chunk 的 Chunk 服务器会变位热点
      • 复制多份此类数据
      • 允许客户端之间彼此分享此类数据
  • 优点：
    • 它减少了客户端和 Master 节点通讯的需求
    • 客户端能够对一个块进行多次操作，这样就可以过与 Chunk 服务器保持较长时间的 TCP 连接来减少网络负载
    • 选用较大的 Chunk 尺寸减少了 Master 节点需要保存的元数据的数量

元数据

所有元数据保存在内存中。

  • 文件和 Chunk 的命名空间
    • 会被持久化到日志文件（本地磁盘和远程服务器）
  • 文件和 Chunk 的对应关系
    • 会被持久化到日志文件（本地磁盘和远程服务器）
  • 每个 Chunk 副本的存放地点
    • 不会被持久化，master 启动或有新的 chunkserver 加入时，master 轮询 chunkserver 获得

为什么分布式存储很难？

  • 高性能 –> 跨越多台服务器的分片数据
  • 大量的服务器 -> 持续的故障
  • 容错 -> 复制
  • 复制 -> 潜在的不一致性
  • 一致性 -> 低性能

为了一致性我们应该怎么做？

  • 理想模型：与单个服务器相同的行为

  • 服务器使用磁盘存储

  • 服务器一次执行一个客户端操作（即使在并发的情况下）

  • 读取操作可以正确反映之前的写入操作

    • 即使服务器崩溃或重启

  • 这样：

    答案：无论 1 还是 2，两者必须看到相同的值

  • 这是一个强一致性模型

  • 但是单个服务器的容错能力很差

为了增强容错性的复制使得很难保持一致性

  • 一个简单但是糟糕的复制方案：
    • 两个复制服务器，S1 和 S2
    • 客户端并发地写两者
    • 客户端只从其中一个读
  • 在我们的例子中，C1 和 C2 的写消息可能按照不同的顺序到达两个复制服务器
    • 如果 C3 读 S1，它可能看到 x=1
    • 如果 C4 读 S2，它可能看到 x=2
  • 或者 S1 收到了写消息，但是向 S2 发出写消息之前客户端崩溃了
  • 这不是强一致的
  • 更好的一致性通常需要复制服务器通过交流来进行同步 – 可能很慢
  • 在性能和一致性之间有很多权衡，我们今天将会学习

情境

  • 许多 Google 服务需要大型的快速统一存储系统 Mapreduce，crawler/indexer，日志存储/分析，Youtube（？）
  • 全局（在单个数据中心上）：任何客户端都可以读取任何文件，允许在应用程序之间共享数据
  • 在许多服务器/磁盘上自动“分片”每个文件以实现并行性能来增加可用空间
  • 自动从故障中恢复
  • 每次部署仅一个数据中心
  • 只是 Google 的应用程序/用户
  • 旨在顺序访问大文件；read or append，即不是针对小对象的低延迟数据库

这东西在 2003 年有什么新鲜的？他们的文章为什么能被 SOSP 收录？

  • 不是分布式，分片，容错的基本思想。

客户端 C 读取文件的步骤是什么？

  1. C 发送文件名和偏移量给 master M（如果没有缓存的话）
  2. M 根据偏移量找到 chunkhandle
  3. M 仅使用最新版本的 chunkserver 列表进行回复
  4. C 缓存 handle + chunkserver list
  5. C 根据 chunkhandle 和 偏移量，向最近的 chunk server 发送请求
  6. chunk server 从磁盘上的 chunk 文件读取数据并返回

master 如何知道哪些块服务器具有给定的块？

C 做一个 “record append” 的步骤？

  1. C向M询问文件的最后一块
  2. 如果 M 看到块没有 primary （或租约已过期（lease expired））：
    1. 如果没有带有最新版本号的chunkserver，则报错
    2. pick primary P and secondaries from those w/ latest version #
    3. increment version #, write to log on disk
    4. tell P and secondaries who they are, and new version #
    5. replicas write new version # to disk
  3. M tells C the primary and secondaries
  4. C sends data to all (just temporary...), waits
  5. C tells P to append
  6. P checks that lease hasn't expired, and chunk has space
  7. P picks an offset (at end of chunk)
  8. P writes chunk file (a Linux file)
  9. P tells each secondary the offset, tells to append to chunk file
  10. P waits for all secondaries to reply, or timeout secondary can reply "error" e.g. out of disk space
  11. P tells C "ok" or "error"
  12. C retries from start if error

GFS 向客户提供什么一致性保证？

  • 需要采用一种告诉应用程序如何使用 GFS 的形式。
  • 这是一种可能性：
    • 如果 primary 告诉客户端记录添加成功，则随后打开该文件并对其进行扫描的任何 reader 都将在某处看到该附加记录。
    • （并不是说失败的追加将不可见，而是对于所有 reader 将看到相同的文件内容或相同的记录顺序。）

对于 GFS 如何满足此保证，我们该如何理解？

  • 看一下它对各种故障的处理：崩溃，崩溃+重启，崩溃+替换，消息丢失，分区。
  • 询问 GFS 如何确保关键属性。
    • 如果附加客户端在尴尬的时刻失败了怎么办？这个尴尬的时刻存在吗？
    • 如果 appending client 已缓存过时的（错误的）primary 怎么办？
    • 如果阅读客户端缓存了过时的 secondary list 怎么办？
    • 主机崩溃+重启会导致它忘记文件吗？还是会忘记了哪些块服务器保存了相关的块？
    • 两个客户确实在同一时间记录追加。他们会覆盖彼此的记录吗？
    • 假设一个 secondary 从未接收到来自 primary 的 append 命令。如果 reading client 从 secondary 读会怎样？
    • 如果 primary 在给所有 secondary 发送 append 命令之前奔溃会怎样？没有看到 append 的 secondary 能被选为新的 primary 吗？
    • chunkserver S4 带着一个过时的 chunk 拷贝离线了。primary 和 所有活跃的 secondaries 崩溃了。S4 在 primary 和 secondaries 之前活过来了。master 可以选择 S4 （带着过时 chunk）作为 primary 吗？有一个带着过时数据的 primary 好，还是根本没有拷贝好？
    • 如果一个 secondaries 总是写失败（例如死机，磁盘空间不够，磁盘损坏） primary 该怎么办？primary 应该从 secondaries 里排除此 secondary ，然后返回给客户端成功吗？还是 primary 应该继续发送操作并使其失败，从而使每个客户端写请求都失败？
    • 如果 S1 存活并且可以接受客户端的请求，但是 master 和 primary 之间的网络通信失败怎么办？“network partition”。master 应该选择一个新的 primary 吗？将会有两个 primary 吗？这样 append 可以发往其中一个，然后再向另一个同步？从而破坏一致性保证？“split brain”
    • 如果一个 partitioned primary 服务了客户端的 appends，同时它的租约过期了，并且 master 选择了一个新的 primary，新的 primary 需要通过 partitioned primary 更新到最新数据吗？
    • 如果 master 崩溃了。代替者知道死机的 master 所知道的吗？（每个 chunk 的版本号，primary，租约到期时间）
    • 谁/什么决定主机已死，必须更换？如果没有响应，master replicas 可以对 master 执行 ping 操作吗？
    • 如果整栋建筑停电怎么办？然后恢复电源，并重新启动所有服务器。
    • 或许因为副本太少。master 想要创建一个新的 chunk 副本。假设它是文件中的最后一块，而且正在被 append。新副本如何确保不丢失任何 append？毕竟它还不是 secondaries 中的一个。
    • GFS 是否会在任何情况下破坏担保？也就是说，append 成功，但后续的reader 看不到该记录。所有主副本永久丢失状态（永久磁盘故障）。可能更糟：结果将是“没有答案”，而不是“不正确的数据”。"fail-stop"。持有该块的所有块服务器将永久丢失磁盘内容。再次，"fail-stop"；不是最糟糕的结果 CPU，RAM，网络或磁盘产生错误的值。校验和可以捕获某些情况，但不是全部。时间未正确同步，因此租约无法解决。因此，多个 primary，也许写到一个，读到另一个。
    • GFS允许哪些可见的应用程序异常行为？所有客户端都会看到相同的文件内容吗？一位客户可以看到另一位客户根本看不到的记录吗？如果客户端两次读取文件，客户端会看到相同的内容吗？所有客户都会以相同的顺序看到成功附加的记录吗？
    • 这些异常会给应用带来麻烦吗？ MapReduce？

如何做到没有异常 – 严格一致性？

也就是说，所有的客户端将会看到相同的文件内容。

给出一个真正的答案太难了，这里有一些建议。

  • primary 应该检测客户端的重复写请求。或者客户端根本就不这样做。
  • 所有的 secondaries 都应该完成每个写，否则什么也不做。试探性地写直到所有人都答应完成？在所有人都同意执行写操作之前，不要暴露它们！
  • 如果 primary 崩溃，复制们将会错失一些操作。新的 primary 必须和所有的复制通信以发现所有最近的操作，并和 secondaries 同步。
  • 为避免客户端从过时的 secondary 中读取内容，要么所有客户端读都必须从 primary 中读，要么 secondary 也必须具有租约。
  • 你将在实验2和3中看到所有这些内容！
