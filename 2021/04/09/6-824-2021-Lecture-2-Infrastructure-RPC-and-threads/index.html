6.824 2021 Lecture 2: Infrastructure: RPC and threads

Posted on 2021-04-09

6.824 2021 Lecture 2: Infrastructure: RPC and threads

Today

Go 里的线程和 RPC，着眼于实验

why Go？

  • 良好的线程支持
  • 便捷的 RPC
  • 类型安全
  • 垃圾收集（GC）
  • 相对简单
  • tour 之后要看 effecive_go

Threads

  • 有用结构化工具虽然有坑

  • Go 称之为 goroutines，其他人称之为线程

    goroutines 和线程的区别：

    • 动态栈，2 KB ~ 1GB
    • 调度，goroutines 在用户态完成m:n（在 n 个系统线程上调度 m 个 goroutine ）调度
    • GOMAXPROCS 变量决定会有多少个操作系统线程来执行 goroutine
    • goroutine 没有 ID 号

  • 线程允许程序同时做许多事情

  • 每个线程串行执行，就像普通的非线程程序

  • 线程共享内存

  • 线程独有的状态：程序计数器，寄存器，栈

Why threads？

  • 线程具有并发性 —— 分布式系统必备

  • IO 并发

    • 客户端并行的向多个服务端发送请求并等待响应
    • 服务端处理多个客户端的请求；请求会阻塞
    • 为处理客户端 X 的请求因从磁盘读取数据而阻塞的同时处理客户端 Y 的请求

  • 多核性能

    • 在多核上并行的运行代码

  • 方便

    • 在后台运行一个线程检查是否每个工作线程还活着

有什么线程的代替品吗？

  • 在单个线程中编写活动（activities）显式交替运行的代码 —— 通常称作“事件驱动”
  • 每个活动（activity）保持一个状态表，例如，客户端请求
  • 一个“事件”循环：
    • 检查每个活动上是否有新的输入（例如，服务端相应到达）
    • 活动进行到下一步
    • 更新状态
  • 事件驱动具有 I/O 并发性
    • 消除了线程的巨大开销
    • 但是没有用到多核性能
    • 编程困难

Threading challenges:

  • 共享数据
    • 例如：两个线程同时做 n = n + 1 ？或者一个线程读取的同时另一个执行加操作？
    • 这就是竞争（race） —— 通常是一个 bug
      • 加锁 sync.Mutex
      • 避免访问共享可变数据
  • 线程同步
    • 例如：一个线程生产数据，另一个线程消费数据，消费者如何挂起（wait）？生产者如何唤醒消费者？
      • channels, sync.Cond, WaitGroup
  • 死锁
    • 互斥，请求和保持，不剥夺，循环等待，（sync.Mutex, channels）

爬虫程序

什么时候使用锁+共享内存，什么时候使用channels？

  • 大多数问题两者都可以
  • 更多的依赖于程序员怎么想
    • 状态 – 锁+共享内存
    • 通信 – channels
  • 对于实验
    • 状态 – 锁+共享内存
    • 阻塞/通知 – sync.Cond / channels

Remote Procedure Call（RPC）

  • 分布式系统最主要的机制；所有实验都要用到 RPC
  • 目的：使 客户端/服务端 通信更容易编程
  • 隐藏网络协议细节
  • 将数据（strings， arrays， maps， &c）转换为“wire format”（可以在线缆上传输的格式，二级制格式）

软件结构

各部分详细职责
  1. RpcServer
    负责导出（export）远程接口
  2. RpcClient
    负责导入（import）远程接口的代理实现
  3. RpcProxy
    远程接口的代理实现
  4. RpcInvoker
    客户方实现：负责编码调用信息和发送调用请求到服务方并等待调用结果返回
    服务方实现：负责调用服务端接口的具体实现并返回调用结果
  5. RpcProtocol
    负责协议编/解码
  6. RpcConnector
    负责维持客户方和服务方的连接通道和发送数据到服务方
  7. RpcAcceptor
    负责接收客户方请求并返回请求结果
  8. RpcProcessor
    负责在服务方控制调用过程，包括管理调用线程池、超时时间等
  9. RpcChannel
    数据传输通道

例子

RPC 面临的问题：如何处理失败？

例如：丢包，断网，slow server，服务器崩溃

客户端失败的状态

  • 客户端将不会收到服务端的回复
  • 客户端将不知道服务端是否收到请求
    • 或许服务端从未看到请求
    • 或许服务端执行了但是崩溃了
    • 或许服务端执行了但是在返回时断网了

最简单的故障处理方法：“best effort” 尽力而为

Call() 等待一段时间，若没有收到回复则重发请求，若几次之后仍为成功则返回错误。

best effort OK 吗？

  • 只读操作
  • 如果重复执行则什么也不做的操作
    • 例如，DB 检查是否插入成功

更好的 RPC 行为：“at most once” 最多一次

思路：服务端 RPC 代码检测重复请求，返回之前的回复而不是重新运行的回复

如何检测重复请求？

  • 客户端为每一个请求保存一个唯一的 ID

    • 重发的请求使用相同的 ID

  • 服务端

at most once 的难点

  • 两个不同的客户端使用相同的请求 ID 咋么办？
    • 更大的随机数
    • 给每个客户端一个唯一的 ID（IP地址），和请求 ID 做拼接作为最终的请求 ID
  • 服务端最终必须丢弃旧的 RPCs 的信息
    • 什么时候丢弃是安全的呢？
      • 想法：
        • 每个客户端有一个唯一的 ID
        • 每个客户端 RPC 有一个序列号
        • 客户端在每个 RPC 中都包含 “看到所有回复 <= X”
        • 就像 TCP 序列号 #s 和 acks
      • 或者只允许客户端同一时间有一个未完成的 RPC
        • seq + 1 到达后允许服务器删除所有 <= seq 的回复
  • 当最初始的还在运行时如何处理重复请求？
    • 服务器还不知道该回复什么
    • 想法：等待标志；等待或忽略

at most once 的服务端崩溃或者重启？

  • 如果 at-most-once 的信息在内存里，信息就会丢失，从而接受重复请求
  • 或许应该将数据备份到磁盘
  • 或许应该有一个专门的负责复制的从服务器

Go 的 RPC 是 at-most-once 的简单形式

  • Go RPC 不会重发请求
  • 若没有收到请求 Go RPC 会返回错误

exactly once（恰好一次）呢？

无限重试 + 重复检测 + 容错服务
