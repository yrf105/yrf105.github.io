【译】6.824 2020 Lecture 4: Primary/Backup Replication

6.824 2020 Lecture 4: Primary/Backup Replication

今天

  • 主/备复制以实现容错
  • VMware FT案例研究，该构想的一个极端版本

主题是（仍然）容错

  • 提供可用性
  • 尽管服务器和网络出现故障
  • 使用复制

复制可以处理哪些类型的故障？

  • 单个副本的 "fail-stop" 失败
    • 风扇停止工作，CPU 过热并自行关闭
    • 有人绊倒了副本的电源线或网络电缆
    • 软件会注意到它已用完磁盘空间并停止
  • 可能不是硬件缺陷或软件错误或人为配置错误
    • 不会经常 fail-stop
    • 可能相关（即，导致所有副本同时崩溃）
    • 但是，有时可以检测到（例如校验和）
  • 地震或全市停电怎么办？
  • 仅当副本在物理上分开时

复制值得 Nx 花费吗？

两种主要的复制方法：

  • 状态转移
    • Primary replica 执行服务
    • Primary 将[新]状态发送到备份
  • 复制状态机
    • 客户将操作发送给 primary，
      • primary 序列化并发送到备份
    • 所有副本执行所有操作
    • 如果有相同的开始状态，相同的操作，相同的顺序，确定性，最终会有相同的结束状态。

状态转移更简单

  • 但是状态可能很大，无法通过网络传输

复制状态机通常产生较少的网络流量

  • 与状态相比，操作通常很小
  • 但是复杂
  • VM-FT使用复制状态机，实验 2/3/4 也是如此

大问题：

  • 要复制什么状态？
  • primary 是否必须等待备份？
  • 什么时候切换到备份？
  • 转换时是否可见异常？
  • 如何加快替换备份的速度？

我们希望副本在什么级别上完全相同？

  • 应用状态，例如数据库的表？
    • GFS以这种方式工作
    • 可以高效；主服务器仅将高级操作发送到备份
    • 应用程序代码（服务器）必须了解容错能力，例如前向操作流
  • 机器级别，例如寄存器和 RAM 内容？
    • 可能允许我们复制任何现有的服务器而无需修改！
    • 需要转发机器事件（中断，DMA和＆c）
    • 需要“机器”修改才能发送/接收事件流...

今天的文章（VMware FT）复制了机器级别的状态

  • 透明：可以运行任何现有的操作系统和服务器软件！
  • 对客户端来说就像是一台服务器

概述

  • [图：应用程序，操作系统，下方的 VM-FT，磁盘服务器，网络，客户端] 单词：
  • hypervisor == monitor == VMM (virtual machine monitor)
  • O / S + app是在虚拟机中运行的 "guest"
  • 两台机器，primary and backup
    • primary 将所有外部事件（客户端数据包＆c）发送到网络上以进行备份"logging channel"，携带日志条目
  • 通常，backup 的输出会被 FT 抑制
  • 如果任一方停止通过网络与对方通话 “goes live” 并提供独家服务
  • 如果 primary goes live，它将停止向备份数据库发送日志条目

VMM 模拟本地磁盘接口

  • 但实际存储在网络服务器上
  • 像客户端一样对待：
    • 通常只有主服务器与磁盘服务器通信（备份的 FT 丢弃）
    • 如果备份 goes live，它将与磁盘服务器通信
  • 外部磁盘使创建新备份的速度更快（不必复制 primary's disk）

主服务器何时必须向备份服务器发送信息？

  • 任何时候发生的可能导致他们发生分歧的事情

  • 任何不是执行指令的确定性结果的东西。

FT 必须处理哪些分歧源？

  • 大多数指令在 primary 和 backup 上执行的方式相同。
    • 我们通过归纳法来假设只要内存+寄存器相同。
  • 来自外部世界的输入-网络数据包。
    • 这些显示为DMA数据加上中断。
  • 中断时间。
  • 不涉及状态功能的指令，例如读取当前时间。
  • 不涉及多核竞赛，因为只有单处理器。

为什么分歧会是一场灾难？

  • backup 的 b / c 状态与 primary 的状态不同，如果 primary 随后崩溃了，客户端将看到不一致的情况。
  • 示例：GFS 租约到期
    • 想象我们正在复制 GFS master
    • 在 60 秒的租约到期之前，Chunkserver 必须发送“请续订”消息
    • 时钟中断驱动 master 的时间观念
    • 假设 chunkserver 在大约60秒左右发送“请续订”
    • 对于 primary，时钟中断会在请求到达后立即发生。master 的 primary copy 将租约续订到同一块服务器。
    • 在备份中，时钟中断在请求之前发生。原版的 backup copy 将使租约到期。
    • 如果 master 发生故障，则备份将接管，它会认为是没有租约的，并将其授予另一个 chunkserver。然后，两个块服务器将具有相同块的租约。
  • 因此：备份必须以相同的顺序，在指令流中的相同点看到相同的事件，

每个日志条目：指令＃，类型，数据。

FT 对定时器中断的处理

  • 目标：primary 和 backup 应在指令流中的同一点处看到中断
  • primary ：
  • FT 设置定时器中断
  • FT 从 CPU 读取指令号
  • FT 在 logging channel 上发送“指令X的计时器中断”
  • FT将中断传递给 primary，然后唤醒它（这取决于CPU支持在第X个指令后中断）
  • backup ：
    • 忽略自己的计时器硬件
    • FT 在 backup 到指令 X 之前看到日志条目
    • FT 告诉 CPU 在指令 X 处中断（至FT）
    • FT 模拟计时器中断给 backup

FT 处理网络数据包到达（输入）

  • Primary：
    • FT 告诉 NIC 将数据包数据复制到 FT 的专用“bounce buffer”中
    • NIC 在某些点会执行 DMA，然后中断
    • FT 收到中断
    • FT 暂停 primary
    • FT 将 bounce buffer 复制到 primary 的内存中
    • FT 在 primary 上模拟 NIC 中断
    • FT 将数据包数据和指令＃发送到备份
  • 备份：
    • FT 从日志流中获取数据和指令＃
    • FT告诉 CPU 在指令 X 处中断（至FT）
    • FT 将数据复制到备份存储器，模拟备份中的 NIC 中断

为什么需要 bounce buffer？

  • 我们希望数据在执行 primary 和 backup 内存中的同一位置出现
  • 否则，它们可能会出现分歧。

请注意，备份必须滞后一个一对一的日志条目 假设primary在指令X之后得到中断或输入 如果备份已在X以后执行，则无法正确处理输入 因此，备份FT在看到第一个日志条目之前根本无法开始执行 然后，它仅执行该日志条目中的指令＃ 并等待下一个日志条目，然后恢复备份

示例：不确定的指令 即使主/备份状态相同，某些指令也会产生不同的结果 例如读取当前时间或周期数或处理器序列号 基本的： FT将CPU设置为在主要执行该指令时中断 FT执行指令并记录结果 将结果和指令＃发送到备份 备份： FT读取日志条目，在指令＃处设置中断 FT然后提供主要参与者获得的价值

输出（发送网络数据包）呢？ 主数据库和备份数据库都执行输出指令 Primary的FT实际上执行输出 备份的FT放弃输出

输出示例：数据库服务器 客户可以发送“增加”请求 DB增加存储的值，并用新值答复 所以： [图表] 假设服务器的值从10开始 网络将客户端请求传递给FT上的主服务器 primary的FT在日志记录通道上发送到备份 FT将请求传递到主数据库和备份数据库 主要执行，将值设置为11，发送“ 11”回复，FT真正发送回复 备份执行，将值设置为11，发送“ 11”答复，FT丢弃 客户端得到一个“ 11”响应，如预期的那样

可是等等： 假设在发送回复后主要崩溃 因此客户得到了“ 11”的回复 AND日志记录通道会丢弃带有客户端请求的日志条目 主要节点已死，因此不会重新发送 备份上线 但它的内存值为“ 10”！ 现在客户端发送另一个增量请求 它将再次获得“ 11”，而不是“ 12” 哎呀

解决方案：输出规则（第2.2节） 在主服务器发送输出之前， 必须等待备份以确认所有先前的日志条目

同样，使用输出规则： [图表] 基本的： 接收客户的“增量”请求 在日志记录通道上发送客户端请求 即将发送“ 11”回复给客户端 首先等待备份以确认先前的日志条目 然后向客户发送“ 11”回复 假设在此顺序中的某个时刻发生了主要的崩溃 如果主服务器在收到备份的确认之前 也许备份没有看到客户的请求，并且没有增加 但主要不会回答 如果主服务器收到备份的确认后 然后客户可能会看到“ 11”回复 但是备份保证已收到客户端请求的日志条目 因此备份将增加到11

输出规则很重要 在所有复制系统中以某种形式发生 严重限制性能 特定于应用程序的智能区域 例如。也许不需要主服务器等待回复只读操作 FT没有应用程序级知识，必须保持保守

问：如果主服务器在从备份获得ACK后立即崩溃， 但是在主要发出输出之前？ 这是否意味着将永远不会生成输出？

答：这是当主数据库发生故障并且备份上线时发生的情况。 备份从主数据库获取一些日志条目。 备份将继续执行那些输出被拒绝的日志条目。 在最后一个日志条目之后，备份将上线-停止丢弃输出 在我们的示例中，最后一个日志条目是客户端请求的到达 因此，在客户端请求到达后，客户端将开始发出输出 因此它将向客户端发出回复

问：但是如果主服务器在发出输出之后崩溃了怎么办？ 备份会在秒时间内发出输出吗？

答：是的。 对于TCP，单击“确定”，因为接收者将忽略重复的序列号。 可以写入磁盘，因为备份会将相同的数据写入相同的块＃。

转换时复制输出在复制系统中非常普遍 客户端需要保持足够的状态以忽略重复项 或设计成复制品无害

问：FT可以应对网络分区吗？它会遭受脑裂吗？ 例如。如果主节点和备份节点都认为对方故障了。 他们俩都会上线吗？

答：磁盘服务器打破了束缚。 磁盘服务器支持原子测试和设置。 如果主服务器或备份服务器认为其他设备已死，请尝试进行测试并设置。 如果只有一个人还活着，它将赢得测试和设置并开始上线。 如果两者都尝试，一个人将会失败并停下来。

磁盘服务器可能是单点故障 如果磁盘服务器已关闭，则服务已关闭 他们可能会想到一个复制的磁盘服务器

问：为什么他们不支持多核？

绩效（表1） 金融时报/非金融时报：令人印象深刻！ 有点慢 记录带宽 直接反映磁盘读取率+网络输入率 对于MySQL，速度为18 Mbit / s 这些数字对我来说似乎很低 应用程序可以每秒至少400兆位的速度读取磁盘 因此，他们的应用程序不是磁盘密集型应用

英国《金融时报》何时会具有吸引力？ 关键但强度较低的服务，例如名称服务器。 其软件不方便修改的服务。

高吞吐量服务的复制又如何呢？ 人们将应用程序级复制状态机用于例如数据库。 状态只是DB，而不是全部的内存和磁盘。 事件是数据库命令（放置或获取），而不是数据包和中断。 结果：更少的细粒度同步，更少的开销。 GFS使用应用程序级复制，如Lab 2＆c一样
