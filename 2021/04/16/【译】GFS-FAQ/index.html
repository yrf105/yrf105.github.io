【译】GFS FAQ

Posted on 2021-04-16

GFS 常见问题

问：为什么 atomic record append at-least-once，而不是 exactly

once？

3.1 节（第 7 步）说，如果在 secondaries 之一上写入失败，则客户端将重试该写入。这将导致在不失败的副本中多次附加数据。尽管有 arbitrary failures（例如，原始请求和客户端重试之间 primary 失败），但不同的设计可能仍可以检测到重复的客户端请求。你将在实验 3 中实现这样的设计，而付出的代价是复杂性和性能。

问：应用程序如何知道块的哪些部分由填充和重复记录组成？

答：为了检测填充，应用程序可以在有效记录的开头放置一个可预测的magic number，或者包括一个仅在记录有效的情况下才有效的校验和。该应用程序可以通过在记录中包含唯一 ID 来检测重复项。然后，如果它读取的 ID 与先前的记录具有相同的 ID，它就知道它们是彼此重复的。 GFS 为处理这些情况的应用程序提供了一个库。

问：考虑到 atomic record append 以不可预测的偏移量将数据写入文件中，客户如何找到他们的数据？

答：Append （通常是 GFS）主要用于顺序读取整个文件的应用程序。此类应用程序将扫描文件以查找有效记录（请参见上一个问题），因此它们无需事先知道记录位置。例如，该文件可能包含一组并发 Web crawlers 遇到的一组链接 URL。任何给定 URL 的文件偏移量都无关紧要；读者只希望能够阅读整个URL集。

问：什么是校验和？

答：校验和算法将一个字节块作为输入，并返回一个数字，该数字是所有输入字节的函数。例如，一个简单的校验和可能是输入中所有字节的总和（和某个大数取余）。 GFS 存储每个块以及该块的校验和。当块服务器在其磁盘上写入块时，它首先计算新块的校验和，然后将校验和与该块一起保存在磁盘上。当块服务器从磁盘读取块时，它还会读取先前保存的校验和，从磁盘读取的块中重新计算校验和，并检查两个校验和是否匹配。如果数据被磁盘损坏，则校验和将不匹配，并且块服务器将返回错误。另外，某些 GFS 应用程序在应用程序定义的记录上方存储了自己的校验和，位于 GFS 文件中，以区分正确的记录和填充。 CRC32 是校验和算法的示例。

问：本文提到引用计数-它们是什么？

答：它们是快照写时复制实现的一部分。当 GFS 创建快照时，它不复制块，而是增加每个块的引用计数器。这使得创建快照的成本不高。如果客户端写入了一个块，并且主服务器注意到引用计数大于1，则主服务器会首先创建一个副本，以便客户端可以更新副本（而不是快照的一部分）。你可以将其视为延迟复制，直到绝对必要。希望不是所有的块都将被修改，并且可以避免生成一些副本。

问：如果应用程序使用标准的 POSIX 文件 API，是否需要进行修改才能使用 GFS？

答：是的，但是 GFS 不适用于现有的应用程序。它是为新编写的应用程序（例如 MapReduce 程序）设计的。

问：GFS 如何确定最近的副本的位置？

答：本文暗示 GFS 会根据存储可用副本的服务器的 IP 地址来执行此操作。在 2003 年，Google 必须以这样一种方式分配 IP 地址：如果两个 IP 地址在 IP 地址空间中彼此靠近，则它们在机房中也彼此靠近。

问：假设 S1 是块的 primary，而 mater 和 S1 之间的网络出现故障。主服务器将注意到并指定其他一些服务器作为 primary，例如 S2。由于 S1 实际上并没有发生故障，因此对于同一块是否存在两个主节点？

答：那将是一场灾难，因为两个 primary 都可能对同一块应用不同的更新。幸运的是，GFS 的租用机制可以避免这种情况。主机授予 S1 60 秒的租约为 primary。当租约到期时，S1 知道不再是 primary。直到先前对 S1 的租约到期，主服务器才会授予 S2 租约。因此，直到 S1 停止后，S2 才开始充当主要对象。

问：对于块大小，64 MB听起来不太好！

答：64 MB 的块大小是 mater 中簿记的单位，并且是在块服务器上分片文件的粒度。客户端可以发出较小的读取和写入操作-他们不是被迫处理整个 64 MB的块。使用如此大的块大小的目的是减小主数据库中元数据表的大小，并避免限制希望进行大量传输以减少开销的客户端。另一方面，小于 64 MB 的文件不会获得太多并行性。

问：Google 仍在使用 GFS 吗？

答：有传言说，GFS 已被称为 Colossus 的东西所取代，它们具有相同的总体目标，但是在 master 性能和容错能力方面有所改进。

问：GFS 以正确性换取性能和简单性的接受程度如何？？

答：这是分布式系统中经常出现的主题。高度的一致性通常需要复杂的协议，并且需要在机器之间进行通信（正如我们将在接下来的几堂课中看到的那样）。通过利用特定类应用程序可以容忍宽松的一致性的方式，人们可以设计出具有良好性能和足够一致性的系统。例如，GFS 针对 MapReduce 应用程序进行了优化，该应用程序需要对大型文件具有较高的读取性能，并且可以在文件中留有空洞，记录显示多次以及读取不一致。另一方面，GFS 不擅长在银行存储帐户余额。

问：如果主服务器出现故障怎么办？

答：有些副本 mater 具有 master 状态的完整副本。该论文的设计需要在 master 故障后进行人工干预才能切换到其中一个副本（第5.1.3节）。稍后，我们将看到如何使用 Raft 构建自动切换到备份的复制服务。

问：为什么有 3 个副本？

答：选择此数字是为了最大程度地减少丢失块的可能性。我想这是根据磁盘的可靠性以及创建新副本所花费的时间计算的。以下是有关那个时代的磁盘可靠性的研究：https://research.google.com/archive/disk_failures.pdf。如果客户端希望文件被复制 3 次以上（例如，允许更多并发读取），则可以指定该文件。

问：有一个单一的 mater 是一个好主意吗？

答：这个想法简化了初始部署，但从长远来看并不是很好。这篇文章-https://queue.acm.org/detail.cfm?id=159420-说随着时间的流逝和 GFS 使用的增加，有些事情出了问题。文件的数量增长使得将所有文件的元数据存储在单个主机的 RAM 中是不合理的。客户数量的增长足以使一个主服务器没有足够的 CPU 能力来为其服务。从发生故障的主机切换到其备份之一需要人工干预，这使恢复速度变慢。显然 Google 用 Colossus 取代了GFS，将主服务器拆分到多个服务器上，并具有更自动化的 master 故障恢复功能。

问：什么是内部碎片？为什么延迟分配有帮助？

答：内部碎片是指系统使用的分配单位大于所请求分配所需的分配单位时所浪费的空间。例如，在 GFS 中，应用程序创建了一个 1 字节的文件，浪费了 64M-1 字节，因为分配大小为 64MB（一块）。 GFS 避免了这个潜在的问题，因为 64MB 是延迟分配的。每个块都是 Linux 文件，因此，当应用程序创建 1 字节文件时，该块的磁盘上表示形式是 1 字节Linux文件。
