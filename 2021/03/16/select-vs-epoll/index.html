select vs. epoll

Posted on 2021-03-16 Edited on 2021-03-17

I/O 多路复用

I/O 多路复用技术使得我们可以在一个进程或线程内实现对多个客户端的数据收发任务，从而避免了进程或线程频繁上下文切换对处理多客户端能力的影响（上下文切换是大头，创建进程以及复杂的 IPC 等也是问题）。

I/O 多路复用是操作系统提供的通知机制，当被监听的文件描述符上注册的事件发生时被阻塞的进程或线程就会被唤醒去执行读写操作。

select 和 epoll 就是两种不同的 I/O 多路复用技术。

使用它们的步骤有相似之处：

  1. 在文件描述上注册事件并加入监听
  2. 调用监听函数阻塞监听文件描述符，当事件发生时监听函数返回发生事件的文件描述符
  3. 对发生事件的文件描述符执行相应的操作。

但是，它们又有许多不同之处。

SELECT

select 编写的回声服务端：

select 存在的问题：

  • 传入 select 函数的文件描述符集合以及超时时间每轮循环都需要赋新值，将文件描述符集从内核态搬到用户态赋新值后在搬回内核态如此往复降低了效率

  • 为了找到是文件描述符集中哪个文件描述符的注册事件发生了需要遍历所有可能的文件描述符，相当于时间复杂度为 O(n)

  • select 监听文件描述符的数量受到常量 __FD_SETSIZE 的限制

epoll

epoll 编写的回声服务端

epoll 解决了 select 的所有问题：

  1. 通过将要监听的文件描述符保存在 epollfd 所指向的文件中，从而避免了每轮循环都在用户空间和内核空间之间搬运文件描述符
  2. 在内核空间中 epoll 通过红黑树来管理 epollfd 所指向文件中的描述符，当被监听的文件描述上有事件发生时，epoll 将有事件发生的文件描述符放在双向链表中，并通过放入事先给定的 events 内存中来返回用户空间，所以 events 数组中的每个文件上都有事件发生，相当于时间复杂度为 O(1)
  3. epoll 能监听的文件描述符限制只与系统资源有关
